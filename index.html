📊 User Flow — Aplikasi Keuangan UMKM

Dokumentasi interaktif seluruh alur fungsional dari perancangan sistem keuangan berbasis cloud untuk UMKM. Klik setiap langkah untuk melihat detail teknis.

🎯 Use Case Diagram

Gambaran umum aktor dan kemampuan masing-masing dalam sistem.

Umum Owner & Karyawan

  • Registrasi (khusus Owner)
  • Login & Logout
  • Mengelola profil pribadi

Owner Pemilik Bisnis

  • Membuat & mengelola profil perusahaan (Multi Perusahaan)
  • Mengundang & memberi hak akses karyawan (Multi Karyawan)
  • Mengelola Data Master (pelanggan, vendor, bagan akun)
  • Mengakses & mengunduh laporan keuangan penuh

Karyawan Kasir / Admin

  • Membuat & mengirim tagihan (invoice)
  • Mencatat penerimaan pembayaran
  • Mencatat pengeluaran operasional

Sistem Backend

  • Validasi token JWT pada setiap request API
  • Membuat Jurnal Umum otomatis (double entry)
  • Rollback jika terjadi kegagalan (prinsip ACID)

🔐 User Flow 1 — Registrasi & Inisialisasi

Alur dari pendaftaran pengguna baru hingga masuk ke dasbor utama.

Tahap 1 Pendaftaran Akun (Register)

Pengguna mengisi formulir registrasi dan menekan tombol "Daftar".

Aksi Pengguna:
  • Membuka halaman registrasi
  • Mengisi: Nama Lengkap, Email, Kata Sandi
  • Menekan tombol "Daftar"
Tugas Backend:
  • Spring Boot menerima permintaan POST /api/v1/auth/register
  • Kata sandi dienkripsi menggunakan Bcrypt
  • Data disimpan ke tabel users

Tahap 2 Masuk Aplikasi (Login)

Pengguna memasukkan kredensial untuk mendapatkan token JWT.

Aksi Pengguna:
  • Membuka halaman login
  • Mengisi Email dan Kata Sandi
  • Menekan tombol "Masuk"
Tugas Backend:
  • Spring Boot memvalidasi kredensial via POST /api/v1/auth/login
  • Jika valid → membuat Token JWT
  • Token dikirim ke klien sebagai tiket identitas untuk API selanjutnya

Tahap 3 Pengecekan Status Perusahaan (Routing Cerdas)

Sistem mengecek apakah pengguna sudah memiliki profil bisnis.

Tugas Backend (Otomatis):
  • Setelah login, frontend memanggil endpoint profil pengguna
  • Backend memeriksa tabel company_roles dan companies
  • Jika belum punya perusahaan → arahkan ke halaman "Buat Perusahaan Baru"
  • Jika sudah punya → langsung ke Dasbor Utama

Tahap 4 Pembuatan Perusahaan (Onboarding)

Pengguna mengisi profil usaha dan sistem menjalankan transaksi ACID.

Aksi Pengguna:
  • Mengisi: Nama Bisnis, Alamat, Nomor NPWP (opsional)
  • Menekan tombol "Simpan Profil"
Tugas Backend (Krusial — @Transactional):
  • Operasi 1: Simpan profil ke tabel companies
  • Operasi 2: Ikat pengguna ke tabel company_roles dengan status OWNER
  • ⚠️ Jika operasi 2 gagal → operasi 1 otomatis dibatalkan (Rollback / ACID)

Tahap 5 Mengakses Dasbor Utama

Pengguna berhasil masuk ke dasbor. Semua data difilter per perusahaan.

Tugas Backend:
  • Setiap query database selalu difilter berdasarkan company_id aktif
  • Ini mengunci keamanan agar data antar pengguna tidak bocor (Multi-Tenant Isolation)
  • Dasbor menampilkan: total saldo kas, tagihan belum dibayar, ringkasan laba rugi

👥 User Flow 2 — Manajemen Karyawan (RBAC)

Alur penambahan karyawan oleh Owner dan sistem kontrol akses berbasis peran.

Tahap 1 Penambahan Karyawan (Oleh Owner)

Owner membuka menu Manajemen Tim dan mengisi data karyawan baru.

Aksi Owner:
  • Masuk ke menu Manajemen Tim
  • Menekan tombol "Tambah Karyawan"
  • Mengisi: Nama, Email, Kata Sandi Sementara, dan memilih Peran (ADMIN / KASIR)
  • Menekan tombol "Simpan"

Tahap 2 Validasi & Penyimpanan (Backend)

Backend melakukan sanitasi input, enkripsi kata sandi, dan penyimpanan transaksional.

🛡️ Keamanan — Sanitasi & Enkripsi:
  • Backend melakukan sanitasi input untuk mencegah XSS dan SQL Injection
  • Kata sandi sementara dienkripsi dengan Bcrypt
🔒 Transaksi ACID (@Transactional):
  • Operasi 1: Simpan data ke tabel users
  • Operasi 2: Simpan relasi ke tabel company_roles (dengan company_id Owner dan peran yang dipilih)
  • ⚠️ Jika salah satu gagal → seluruh proses dibatalkan

Tahap 3 Login Karyawan

Karyawan baru masuk menggunakan kredensial yang diberikan Owner.

Aksi Karyawan:
  • Membuka aplikasi dan login dengan Email + Kata Sandi Sementara
Tugas Backend:
  • Validasi kredensial → terbitkan Token JWT
  • Token JWT karyawan berisi informasi spesifik: Role: KASIR dan company_id
  • Berbeda dengan token Owner yang memiliki akses penuh

Tahap 4 Otorisasi Akses (Spring Security)

Sistem menjaga endpoint berdasarkan peran pengguna. Akses ditolak jika tidak berwenang.

Contoh Skenario:
  • Karyawan (KASIR) mencoba akses halaman Laporan Keuangan
  • Spring Security membaca token JWT → peran = KASIR
  • Endpoint laporan hanya boleh diakses OWNER atau ADMIN
  • ❌ Hasil: HTTP 403 Forbidden (Akses Ditolak)
Skenario Diizinkan:
  • Karyawan (KASIR) mengakses endpoint Pembuatan Invoice
  • ✅ Hasil: Akses diizinkan, kasir bisa membuat tagihan

📄 User Flow 3 — Pencatatan Keuangan (Tagihan & Jurnal)

Alur pembuatan tagihan, pembayaran, dan pembuatan jurnal akuntansi secara otomatis.

Tahap 1 Pembuatan Tagihan (Invoice)

Kasir atau Admin membuat tagihan baru untuk pelanggan.

Aksi Pengguna:
  • Masuk ke menu Tagihan → tekan "Buat Tagihan Baru"
  • Memilih kontak pelanggan (dropdown pencarian cepat)
  • Memasukkan detail: Deskripsi barang/jasa, Jumlah, Harga Satuan
  • Menekan "Simpan Tagihan"

Tahap 2 Validasi & Penyimpanan Tagihan (Backend)

Backend menghitung ulang total harga dan menyimpan data tagihan.

🛡️ Keamanan — Presisi Data:
  • Backend wajib menghitung ulang total harga dari setiap baris barang di sisi server
  • Mencegah manipulasi nominal harga dari frontend
  • Menggunakan tipe NUMERIC(19,4) untuk menghindari error pembulatan
Penyimpanan:
  • Data induk → tabel invoices (status: BELUM LUNAS)
  • Detail barang → tabel invoice_items

Tahap 3 Pencatatan Pembayaran (Oleh Kasir)

Kasir mencatat pembayaran dari pelanggan dan memilih akun tujuan.

Aksi Kasir:
  • Membuka detail tagihan yang belum lunas
  • Menekan tombol "Terima Pembayaran"
  • Memasukkan Nominal pembayaran
  • Memilih akun tujuan: Kas Kecil atau Rekening Bank BCA

Tahap 4 Eksekusi Jurnal Ganda & ACID (Backend Krusial)

Backend menjalankan tiga operasi database secara transaksional dengan prinsip double entry.

🔒 Transaksi ACID (@Transactional) — Titik Paling Kritis:
  • Operasi 1: Perbarui status invoices → LUNAS
  • Operasi 2: Buat satu baris referensi di tabel journal_entries
  • Operasi 3: Buat dua baris di tabel journal_lines:
📒 Logika Jurnal Ganda (Double Entry):
  • Debit: Akun "Rekening Bank BCA" (uang bertambah)
  • Kredit: Akun "Pendapatan Usaha" (pendapatan tercatat)
⚠️ Jaminan Keamanan:
  • Jika terjadi error, server mati, atau koneksi terputus → Rollback total
  • Status tagihan kembali ke BELUM LUNAS
  • Uang tidak tercatat secara sepihak

🔌 Perancangan API Endpoint

Daftar lengkap endpoint REST API yang dirancang untuk sistem keuangan.

A. Autentikasi & Pengguna

MethodEndpointKeterangan
POST/api/v1/auth/registerMendaftarkan pengguna baru
POST/api/v1/auth/loginLogin pengguna, mengembalikan token JWT

B. Manajemen Perusahaan (Tenant)

MethodEndpointKeterangan
POST/api/v1/companiesMembuat profil perusahaan baru
GET/api/v1/companiesMelihat daftar perusahaan yang dapat diakses user

C. Manajemen Transaksi (Invoice)

MethodEndpointKeterangan
POST/api/v1/invoicesMembuat tagihan baru beserta items
GET/api/v1/invoicesMengambil daftar invoice per perusahaan (paginasi)
PUT/api/v1/invoices/{id}/payMemproses pembayaran → otomatis buat jurnal

D. Modul Pelaporan (Reports API)

MethodEndpointKeterangan
GET/api/v1/reports/general-journalMengambil Jurnal Umum
GET/api/v1/reports/general-ledger/{accountId}Mengambil Buku Besar per akun
GET/api/v1/reports/trial-balanceMengambil Neraca Saldo
GET/api/v1/reports/profit-and-lossMengambil ringkasan Laba Rugi
GET/api/v1/reports/balance-sheetMengambil struktur Neraca
GET/api/v1/reports/cash-flowMengambil Arus Kas (filter akun Kas/Bank)
⚠️ Semua endpoint laporan wajib menggunakan filter parameter waktu: ?startDate=2024-01-01&endDate=2024-01-31

🏗️ Struktur Proyek Backend (Spring Boot)

Arsitektur kode yang terorganisir untuk kemudahan pengembangan dan perawatan.

src/main/java/com/flow/keuangan/ ├── config/ ← Konfigurasi Spring Security, JWT Auth, CORS ├── controller/ ← Menerima request HTTP, memanggil service, mengembalikan JSON ├── service/ ← Jantung logika bisnis: validasi, kalkulasi, jurnal double entry ├── repository/ ← Interface Spring Data JPA untuk komunikasi database ├── entity/ ← Class Java merepresentasikan tabel database ├── dto/ ← Data Transfer Object — jangan ekspos entity langsung ke klien └── exception/ ← Global Exception Handler (@ControllerAdvice)

🐛 Penanganan Error Global

🎯 @ControllerAdvice

  • Semua error ditangkap di satu titik pusat
  • Data tidak ditemukan, akses ditolak, validasi gagal

📋 Format Respons Standar

  • JSON seragam: timestamp, status, message, errorCode
  • Tidak membocorkan stack trace server

📝 Pencatatan Log

  • Error krusial dicatat via SLF4J
  • Log internal, tidak ditampilkan ke pengguna

🎨 Prinsip Desain UI/UX

📊 Dasbor Utama

  • Grafik total saldo kas
  • Daftar tagihan belum dibayar
  • Ringkasan laba rugi bulan berjalan

📝 Formulir Transaksi

  • Dropdown pencarian cepat untuk kontak & akun
  • Perhitungan total otomatis di layar
  • Intuitif tanpa kalkulator terpisah

📱 Responsivitas

  • Adaptif untuk komputer dan ponsel
  • Frontend: React/Vue + Axios
  • Manajemen sesi JWT otomatis
